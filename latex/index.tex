This repository hosts an optimizer for Fully Homomorphic Encryption (FHE) working on an AST-\/like (Abstract Syntax Tree) representation of a program. In future, this optimization component will be integrated into the \href{https://github.com/MarbleHE/Marble}{\texttt{ Marble}} FHE compiler \mbox{[}1\mbox{]}.

\hypertarget{index_autotoc_md1}{}\doxysubsection{Optimizations}\label{index_autotoc_md1}
The optimizations this tool aims to support can be divided into the following two groups\+:

{\bfseries{1. AST-\/to-\/\+Circuit Transformations.}} These transformations take an AST that is constructed manually or derived from code, and then transforms the AST into a circuit by transforming non-\/compatible operations (e.\+g., If-\/ and While-\/\+Statements) into their circuit-\/equivalent using gates.

{\bfseries{2. Circuit-\/to-\/\+Circuit Transformations.}} These transformations take a circuit as input and transform it into an equivalent optimized circuit that allows performing FHE computations more efficiently. \mbox{\hyperlink{class_for}{For}} example, by rewriting the circuit to reduce the multiplicative depth. This allows using smaller parameters that, in turn, enable more efficient computation.\hypertarget{index_autotoc_md2}{}\doxyparagraph{Compile-\/\+Time Expression Simplifier}\label{index_autotoc_md2}
The optimizer contains a compile-\/time expression simplifier that leverages existing knowledge (e.\+g., known variable values) to simplify the AST. The goal of this step is to generate an AST that is more easily transferable to a circuit. \mbox{\hyperlink{class_for}{For}} example, the simplifier removes all variable declarations and assignments by replacing the respective variable uses with the variable\textquotesingle{}s value. Also, operations that are not circuit-\/compatible such as \mbox{\hyperlink{class_if}{If}} statements are rewritten. \mbox{\hyperlink{class_for}{For}} example, an \mbox{\hyperlink{class_if}{If}} statement of the form 
\begin{DoxyCode}{0}
\DoxyCodeLine{IF (condition) \{ x = 5; \} ELSE \{ x = 21; \};}

\end{DoxyCode}
 can be rewritten as 
\begin{DoxyCode}{0}
\DoxyCodeLine{x = (condition)*5 + (1-\/condition)*21;}

\end{DoxyCode}
 without using any \mbox{\hyperlink{class_if}{If}} statement. Furthermore, the simplifier evaluates parts of the computation that are known at compile time, e.\+g., arithmetic or logical expressions.\hypertarget{index_autotoc_md3}{}\doxyparagraph{Cone-\/\+Rewriting}\label{index_autotoc_md3}
Our optimizer implements the heuristic approach for minimizing the multiplicative depth in Boolean circuits, as proposed by Aubry et al. \mbox{[}2\mbox{]}. This technique finds and rewrites certain structures of a circuit (so-\/called {\itshape cones}) in a way that the multiplicative depth is decreased locally. Repeating this procedure for all cones on the critical path leads to a global reduction of the multiplicative depth.

The approach uses four different algorithms\+: An algorithm (Alg. 1) for cone construction that takes a Boolean circuit and a minimum multiplicative depth, and returns the set delta Δ of all reducible cones. This set is taken by Alg. 2 to create a circuit C$^\wedge$\{AND\} consisting of all critical AND nodes. Thereafter, a flow-\/based algorithm (Alg. 3) takes this C$^\wedge$\{AND\} circuit to determine Δ$^\wedge$\{MIN\}, the minimum set of cones that are required to reduce the multiplicative depth. It is desirable to minimize the number of cones because each cone rewriting adds additional nodes to the circuit. Lastly, Alg. 4 rewrites the minimum set of reducible cones while ensuring that the multiplicative depth reduced. The whole procedure is repeated until there are no reducible cones found anymore.

The implementation is heavily based on the pseudo-\/codes provided in the paper, except Alg. 2 (C$^\wedge$\{AND\} construction) for which no pseudo code was given. Deviant from the paper\textquotesingle{}s implementation (see Sect. 3.\+3), we also did not consider yet the case in which no more reducible cones are available in C$^\wedge$\{AND\} but the multiplicative depth can be reduced further.\hypertarget{index_autotoc_md4}{}\doxyparagraph{CKKS Scheme-\/\+Specific Optimizations}\label{index_autotoc_md4}

Following, the different node types are briefly explained. The examples in brackets show how the commands would look like in \char`\"{}plain\char`\"{} C++.


\begin{DoxyItemize}
\item Classes derived from {\ttfamily \mbox{\hyperlink{class_abstract_expression}{Abstract\+Expression}}}
\begin{DoxyItemize}
\item {\ttfamily Binary\+Expr} – a binary arithmetic expression (e.\+g., {\ttfamily 13 + 37}).
\item {\ttfamily \mbox{\hyperlink{class_call}{Call}}} – a call to an internal function, i.\+e., its implementation is represented in the AST as a \mbox{\hyperlink{class_function}{Function}}.
\item {\ttfamily Call\+External} – a call to an external function which is treated as {\itshape black box}, i.\+e., without accessibility to its definition.
\item {\ttfamily \mbox{\hyperlink{class_function_parameter}{Function\+Parameter}}} – describes the parameters that a function accepts. To evaluate an AST, values must be passed for each of the parameter defined by the function\textquotesingle{}s {\ttfamily \mbox{\hyperlink{class_function_parameter}{Function\+Parameter}}} node.
\item {\ttfamily \mbox{\hyperlink{class_literal}{Literal}}} – base class for all \mbox{\hyperlink{class_literal}{Literal}} derived from.
\item {\ttfamily Literal\+Bool} – models a Boolean value.
\item {\ttfamily Literal\+Int} – models an integer value.
\item {\ttfamily Literal\+String} – models an string value.
\item {\ttfamily Literal\+Float} – models a float value.
\item {\ttfamily Logical\+Expr} – a (binary) logical expression using a logical operator (e.\+g., {\ttfamily z $<$ 42}).
\item {\ttfamily Unary\+Expr} – a unary expression (e.\+g., {\ttfamily !b} where {\ttfamily b} is a Boolean).
\item {\ttfamily \mbox{\hyperlink{class_variable}{Variable}}} – a variable identified by an identifier (name).
\end{DoxyItemize}
\item Classes derived from {\ttfamily \mbox{\hyperlink{class_abstract_statement}{Abstract\+Statement}}}
\begin{DoxyItemize}
\item {\ttfamily \mbox{\hyperlink{class_block}{Block}}} – a code block {\ttfamily \{...\}}, e.\+g., the then-\/clause of an if statement.
\item {\ttfamily \mbox{\hyperlink{class_function}{Function}}} – a function definition.
\item {\ttfamily \mbox{\hyperlink{class_if}{If}}} – an if-\/conditional statement including both a then-\/branch and an else-\/branch (e.\+g., {\ttfamily if (condition) \{ ... \} else \{ ... \}}), or either only a then-\/branch.
\item {\ttfamily \mbox{\hyperlink{class_return}{Return}}} – a return statement of a method (e.\+g., {\ttfamily return} ).
\item {\ttfamily Var\+Assignm} – the assignment of a variable.
\item {\ttfamily Var\+Decl} – a variable declaration (e.\+g., {\ttfamily Z})
\item {\ttfamily While} – a while-\/loop (e.\+g., {\ttfamily while (condition) \{...\}}).
\item {\ttfamily Call\+External} – {\itshape see above.}
\item {\ttfamily \mbox{\hyperlink{class_return}{Return}}} – a return statement of a method, i.\+e., the output of a computation.
\end{DoxyItemize}
\end{DoxyItemize}
